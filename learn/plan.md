## 总体技术架构

### 1. 前端 H5（React + Vite）
负责：
- 房间加入/离开
- 摄像头与麦克风采集
- WebRTC 1v1 视频通话
- 直播观看页/主播页
- AI 通话页
- WebSocket 信令连接
- 后续语音字幕、AI 响应展示

### 2. 信令服务（Node.js + WebSocket）
负责：
- 创建房间
- 用户进出房间广播
- Offer / Answer / ICE Candidate 转发
- 房间内角色管理（caller / callee / host / audience / ai-agent）
- 心跳保活、断线清理

### 3. 媒体层
分两条线：
- **Phase 1**：1v1 视频通话直接用浏览器 WebRTC P2P
- **Phase 2**：直播改为媒体服务器模式，建议使用 **SRS** 作为本地直播服务

### 4. AI 网关层
负责把前端送来的：
- 文本
- 音频转写结果
- 截帧图片
- 后续低帧率视频片段
统一转成模型请求。

这样前端不直接耦合具体模型，后期更换本地模型时只改网关。

## 关于“Qwen Omni / 全模态”路线
你提到 qwen 的 omni 全模态模型。方案上可以兼容，但这里先不绑定具体下载包名，原因是：
- 本地实时视频通话 + 全模态推理对 **显存 / 推理吞吐 / 编解码链路** 要求很高。
- 在你还未安装环境前，先把**代码接口和协议层**设计好更稳妥。

因此本方案采用：
- 前端统一走 `AI Session` 协议
- 后端网关抽象为 `model adapter`
- 以后可接：Qwen Omni / Qwen2-VL / MiniCPM-V / Whisper / 本地 TTS

换言之，**先把工程架构写对，再决定最终本地模型组合**。

## Phase 划分与验收门禁

## Phase 1：局域网 1v1 视频通话基础版

### 目标
完成手机 H5 在局域网内访问电脑服务，支持：
- 打开房间页
- 采集摄像头/麦克风
- 双端加入同一房间
- 建立 WebRTC 视频通话
- 挂断与重连

### 范围
- React + Vite 前端骨架
- Node.js WebSocket 信令服务
- 基础房间管理
- WebRTC P2P 通话
- LAN 配置文档

### 不做
- 直播
- AI 对话
- 外网 NAT 穿透
- 持久化数据库

### 关键设计
- 前端页面：`/join`、`/room/:roomId`
- 信令消息：
  - `join-room`
  - `peer-joined`
  - `offer`
  - `answer`
  - `ice-candidate`
  - `peer-left`
  - `error`
- 房间先限制 **最多 2 人**，降低复杂度
- ICE 先使用浏览器默认 + 预留 `iceServers` 配置字段

### 验证标准
必须全部通过：
1. 电脑浏览器可创建房间并进入。
2. 手机浏览器在同一 WiFi 下可访问并加入同一房间。
3. 两端都能看到对方视频与听到音频。
4. 任一端断开后，另一端收到离开提示。
5. 重进房间后可重新建立连接。

### 建议验证步骤
- 本机双标签页验证
- 同 WiFi 手机真机验证
- 切换前后摄像头验证
- 手动断网/关闭页面验证

### 通过门禁
只有上述验证全部通过，才进入 Phase 2。

## Phase 2：直播能力（局域网直播）

### 目标
在现有通话基础上增加直播能力：
- 主播开播
- 观众观看
- 主播与观众角色分离
- H5 直播页可播放

### 技术选择
建议引入 **SRS**：
- 本地部署简单
- 支持 RTMP / HTTP-FLV / WebRTC
- 后续可平滑扩展低延迟直播

### 范围
- 直播房间模型
- 主播推流控制 UI
- 观众播放页
- 媒体服务配置文档
- 与信令服务联动的房间状态展示

### 架构选择
- 主播端：采集摄像头与麦克风
- 推到本地媒体服务
- 观众端：拉流观看

### 验证标准
1. 主播能开始/停止直播。
2. 局域网另一台手机可进入直播页观看。
3. 观众端延迟可接受，播放稳定。
4. 主播结束直播后，观众端出现结束状态。

### 通过门禁
直播稳定后再进入 Phase 3。

## Phase 3：本地 AI 文本/视觉会话骨架

### 目标
先把 AI 通话的“控制面”和“协议面”做出来，不依赖你现在安装模型：
- AI 会话页
- 与本地 AI 网关建立 WebSocket
- 发送文本消息
- 发送视频截图帧
- 以流式方式接收 AI 回复

### 范围
- `ai-gateway` 项目骨架
- 统一消息协议
- 模型适配器接口
- 前端 AI 对话 UI
- 截帧与发送逻辑
- 文档中列出未来接入的本地模型选项

### 关键抽象
统一 AI 事件：
- `session.start`
- `input.text`
- `input.image`
- `response.delta`
- `response.done`
- `response.error`

### 验证标准
即便模型未安装，也要能完成：
1. 前端能连接本地 AI 网关。
2. 文本消息能发送并收到 mock 流式返回。
3. 截图帧能上传到网关。
4. 网关日志能正确识别会话与输入类型。

### 通过门禁
协议跑通后再进入 Phase 4。

## Phase 4：语音链路（本地语音对话）

### 目标
实现 AI 通话最核心的人机语音链路：
- 前端采集麦克风
- 发送音频片段到本地网关
- 本地语音识别（后续接 Whisper 类）
- 把识别文本送给模型
- 把模型文本响应送给本地 TTS
- 前端播放 AI 语音

### 范围
- 音频采集与编码策略
- 语音消息协议
- STT/TTS 适配器抽象
- 对话状态机（聆听 / 思考 / 播放）

### 验证标准
1. 前端能录音并上传。
2. 网关能返回 mock / 实际转写文本。
3. 前端能播放 AI 返回音频。
4. 整个回合链路可闭环。

### 通过门禁
语音闭环 OK 后再进入 Phase 5。

## Phase 5：视频理解增强与“类全模态通话”

### 目标
在已有文本、图片、语音基础上，增强为接近实时视频理解：
- 定时截帧
- 事件触发截帧
- 低频视频摘要
- AI 根据画面 + 语音 + 文本联合回答

### 注意
这一阶段才真正逼近你说的“全模态视频通话”。
但在纯本地部署下，是否能做到高实时性，取决于你的机器性能。

### 验证标准
1. 前端可定时发送截图。
2. AI 回复能体现画面上下文。
3. 语音问答能引用最近画面内容。
4. 性能指标达到可接受范围。

## 风险与取舍
- 不建议一开始就做“多人直播 + AI 实时视频通话 + 全模态”大一统，因为排障成本太高。
- 先把 **局域网 1v1 通话** 做稳定，是整个系统的根基。
- 本地“全模态”最容易卡在模型性能，不应在前期把工程协议写死在某个模型 API 上。
