# Phase 4 运行与验证指南

## 本阶段产物

在 Phase 3 基础上，同一 AI 会话页 **`/ai`** 增加：

- **开始录音 / 停止并上传**：将麦克风录音以 **`input.audio`** 发往网关
- 网关返回 **`stt.result`** → **`response.delta` / `response.done`** → **`response.audio`（WAV）**
- 前端自动播放 TTS 音频（受浏览器自动播放策略影响时可能需用户曾与页面交互）

协议与设计见 **`docs/phase-4.md`**。

## 先决条件

- 已完成 **Phase 3** 相关服务启动方式（前端 + `ai-gateway`）
- **必须重启过 `ai-gateway`**，确保进程加载含 `input.audio` 的最新代码（旧进程会对 `input.audio` 报 `unknown message type`）
- 浏览器已授权 **麦克风**（与摄像头同属 `getUserMedia` 流，需音频轨可用）

## 启动步骤

与 Phase 3 相同：

1. `learn/app/ai-gateway`：`npm run dev`（默认 **8790**）
2. `learn/app/frontend`：`npm run dev`（默认 **5173**）
3. 打开 **`/ai`**，点击 **连接 AI 网关**

## 验证步骤

### 验证 1：录音与上传

- 点击 **「开始录音」**，说话 1～3 秒。
- 点击 **「停止并上传」**。
- 对话区出现 **「（语音输入）」** 用户气泡；随后出现 **转写样式** 气泡（`stt.result` 内容）。

### 验证 2：文本流式与 TTS

- 转写出现后，应继续出现助手 **流式** mock 文本，最后收束为一轮助手气泡。
- 随后应听到 **短提示音**（mock TTS：正弦波 WAV）。若无声音，查看页面是否提示播放失败（部分浏览器对自动播放较严）。

### 验证 3：状态提示

- 上传后至播放结束（或失败）前，**「识别 / 推理 / 合成中…」** 类提示应与 `voiceBusy` 一致。
- 播放结束后回到可再次录音状态。

### 验证 4：与 Phase 3 回归

- 同页再次测试 **文本发送**、**截图发送**，应仍正常。
- 语音处理进行中时若快速发送第二条占线请求，网关可能返回 **`busy`**（符合当前互斥设计）。

### 验证 5（可选）：网关日志

- 控制台应能看到 **`stt.result (mock)`** 及流水线相关日志，无未捕获异常。

## Phase 4 完成标准

1. 前端能录音并上传。
2. 能收到并展示 mock **转写**。
3. 能收到并 **播放**（或明确失败原因）mock **WAV**。
4. 整轮 **STT → 文本 → TTS** 可重复触发，且 Phase 3 能力无回归。

## 常见问题

| 现象 | 可能原因 |
|------|----------|
| `unknown message type: input.audio` | 网关未更新或未重启，仍运行旧版本 |
| 连接失败 | 8790 被占用或防火墙拦截 |
| 无法录音 | 未授权麦克风、或音频轨被关闭 |
| 无 TTS 声音 | 浏览器拦截自动播放；可先点击页面任意按钮再试 |

## 后续替换真实 STT / TTS

保持 **`input.audio` / `stt.result` / `response.audio`** 等消息形态不变，替换 `ai-gateway/src/adapters/` 内实现及 `voice-pipeline.js` 中调用即可；前端仅需在 mime、时长或分片策略上按需调整。
